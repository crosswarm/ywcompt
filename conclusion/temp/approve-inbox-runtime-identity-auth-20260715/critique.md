# 评审结论

- P0：只修旧进程无法解决 401；必须保证服务子进程继承托管代理上下文，并禁止本地凭证回退。
- P0：只按租户过滤不足以隔离同租户不同用户；数据域必须同时包含 Profile、用户和租户。
- P0：同步失败后不得分析或返回旧身份缓存；所有业务接口都需统一身份守卫。
- P1：固定端口交接必须验证实例归属，未知进程不可强杀。
- P1：健康检查必须包含真实 CLI 身份探测，不能以 HTTP 端口可访问代替。
- P0：审批不能只验证 ID membership；相同 ID 的 URL、任务号、业务键、服务编码或租户变化必须阻断危险命令。
- P0（平台边界）：当前 req-proxy 在危险请求发出时读取可变的全局 YonWork auth session，未提供 expected-scope/lease。仓库已补最新任务签名和每命令前后身份验证，但严格原子审批仍需平台契约。
- P0：危险 CLI 的本地参数解析错误不能归类为远端结果未知；必须证明是否开始远端请求。本次真实错误由不受支持的 `--yes` 引起，已删除并增加“请求未发送”的明确分类。
- P0：详情快照过期不能伪装成身份变化；同时也不能为消除报错而把旧详情重绑到新快照。当前实现选择安全骨架 + 当前快照重抓取，附件继续 fail closed。
- P1：部分业务类型的详情完整性仍受 sibling CLI `loadExtend` 元数据解析能力限制；必须显示明确不可用状态，不能回退旧字段或伪造成功。
- 用户已明确批准实施，并将真实 YonWork 环境无工程性 401 设为发布硬门槛。
